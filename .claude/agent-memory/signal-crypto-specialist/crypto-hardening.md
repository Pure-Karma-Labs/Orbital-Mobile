Hardening work from Issues #114, #130. These are hardening measures applied to our own wrapper crate code (attachment_crypto.rs, content_crypto.rs), distinct from [[libsignal-api-learnings]] which covers libsignal's own API quirks.

## Zeroize feature flags and gotchas

- `aes`, `cbc`, `aes-gcm` all support `features = ["zeroize"]` — enables zeroize-on-drop for cipher state.
- `sha2` has NO `zeroize` feature. It is a hash (not keyed), so there is nothing secret to zero. Do not waste time looking for it.
- `hmac` has a `"reset"` feature but this does NOT enable zeroize-on-drop. The `Zeroizing` wrapper on the input key material is the correct approach, not a feature flag on hmac.
- `Zeroizing::new(keys)` must be the FIRST statement after the FFI boundary parameter, BEFORE any length/format validation. This ensures keys are zeroed on all exit paths including early-return errors. Moving validation before `Zeroizing` creates a path where raw key bytes survive on the stack after an error return.

**Why:** Cargo feature flags are not self-documenting. Future contributors may assume "just add zeroize to every crate" and waste time or create false confidence.

**How to apply:** When adding new Rust crypto functions that accept key material, follow the `attachment_encrypt` / `aes_gcm_encrypt` pattern: `let key = Zeroizing::new(key);` on line 1, then validate.

## Constant-time comparison

- `subtle::ConstantTimeEq` (crate `subtle = "2.6"`) for digest comparison in `attachment_decrypt`. Prevents timing side-channels on the SHA-256 digest check.
- HMAC verification uses `mac.verify_slice()` which is already constant-time internally.
- Length check must precede `ct_eq` — `ct_eq` panics on mismatched lengths.

## KAT vectors

- KAT (Known-Answer Test) vectors in `attachment_crypto.rs` were generated by an independent reference implementation (pycryptodome), not by calling our own code. This is critical — self-generated vectors only prove self-consistency, not correctness.
- Three vectors covering: non-empty plaintext, empty plaintext, and block-aligned (16-byte) plaintext that forces a full PKCS7 padding block.
- `attachment_encrypt_inner` (the deterministic-IV variant) is used for KATs because the public API generates random IVs.

## Inner function visibility guard

- `attachment_encrypt_inner(plaintext, keys, iv)` accepts an explicit IV and MUST remain `fn` (not `pub`, not `#[uniffi::export]`).
- If exposed via FFI, a caller could reuse IVs with the same key, which completely breaks CBC confidentiality (identical plaintext blocks produce identical ciphertext blocks).
- This principle generalizes: any function that accepts a nonce/IV parameter for a mode that requires uniqueness (CBC, GCM, CTR) must never be part of the public API surface.

## Opaque error strategy

- All decryption failure modes (MAC mismatch, digest mismatch, padding error, wrong key) return the same `SignalError::InvalidMessage { reason: "decryption failed" }`.
- This prevents padding oracle attacks and other error-oracle information leakage.
- Error messages must NOT include which check failed or what the expected value was.
